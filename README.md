## 1. Core Architecture Specifications

### 1.1 CPU Bit Width

I have defined the system architecture as **16-bit**. This standard balances computational capability with hardware feasibility for a solo development cycle.

* **Data Path:** 16-bit (The CPU processes 2 bytes per cycle).
* **Performance Theory:** Compared to an 8-bit architecture running at an identical clock frequency, this design offers approximately double the data throughput per cycle.
* **Design Constraint:** All General Purpose Registers (GPRs) and internal buses are locked to a 16-bit width.

### 1.2 Endianness

The system utilizes **Big Endian** byte ordering.

* **Definition:** The Most Significant Byte (MSB) is stored at the lowest memory address.
* **Justification:** Big Endian is more intuitive for human readability during memory dumps and debugging (reading hex values left-to-right matches the stored order).

## 2. Memory Architecture & Address Space

### 2.1 Physical Address Space

While the CPU data path is 16-bit (limiting direct addressing to  or 64 KB), I have implemented an extended address bus to support modern software requirements.

| Bus Type | Width | Reach | Note |
| --- | --- | --- | --- |
| **Data Bus** | 16 pins | N/A | Transfers one Word (16-bit) per cycle. |
| **Address Bus** | 24 pins | **16 MB** | Accesses  physical locations. |

### 2.2 Memory Management: Segmentation (Banking)

To bridge the gap between 16-bit registers and the 24-bit physical address space, the CPU implements a **High-Byte Concatenation Unit**.

**Addressing Logic:**
The CPU views memory as a collection of 256 distinct **64 KB Pages** (Segments). A physical address is generated by the Address Unit (AU) by combining:

1. **Segment Register (8-bit):** Selects the "Page" (Base Address).
2. **Offset (16-bit):** Specifies the exact location within that page.

**Mechanism:**



*Note: This architecture introduces hardware-level memory protection. A program running in Segment `0x10` cannot accidentally overwrite data in Segment `0x20` without explicitly changing the Segment Register.*

## 3. Register File Organization

I have designed a streamlined register file to maximize instruction encoding efficiency.

### 3.1 General Purpose Registers (GPR)

The system contains **8 Registers** (R0 – R7).

* **Encoding:** 3 bits are required to address each register ().
* **Layout:**
* **R0–R6:** General Purpose Registers for arithmetic and pointers.
* **R7 (Extension Register):** Reserved for **Widening Operations**. When performing arithmetic that exceeds 16 bits (e.g., multiplication or overflow), R7 captures the "spillover" (high-order bits).

## 4. Instruction Set Architecture (ISA)

### 4.1 Instruction Encoding Format

The system utilizes a **Fixed-Length 16-bit Instruction Format**.

* **Efficiency:** A fixed width simplifies the hardware Control Unit (Fetch/Decode logic) significantly compared to variable-length CISC designs.
* **Structure:** Every instruction is exactly 1 Word (16 bits) long.

### 4.2 Bit Distribution (Three-Operand Layout)

To maintain a powerful instruction set within 16 bits, I have adopted the following encoding scheme:

| Field | Bit Width | Description |
| --- | --- | --- |
| **Opcode** | **7 bits** | Defines the operation (Supports  unique instructions). |
| **Destination (Rd)** | **3 bits** | Target register for the result. |
| **Source 1 (Rs1)** | **3 bits** | First operand register. |
| **Source 2 (Rs2)** | **3 bits** | Second operand register. |

**Total Length:** 

### 4.3 Example Assembly

```asm
ADD R1, R2, R3   ; R1 = R2 + R3

```

* **Opcode (ADD):** 7 bits
* **Rd (R1):** 001
* **Rs1 (R2):** 010
* **Rs2 (R3):** 011

## 5. Memory Mapping (ROM / RAM / I/O)

With a **16 MB** physical address space ($000000 – $FFFFFF), the memory map is strictly partitioned to ensure predictable I/O and boot behavior.

**Reset Vector:**
Upon power-up, the Program Counter (PC) initializes to `$000000` (ROM Start).

### 5.1 System Memory Map Layout

| Address Range (Hex) | Size | Component | Description |
| --- | --- | --- | --- |
| **$000000 – $00FFFF** | 64 KB | **ROM** | Firmware, BIOS, Bootloader. (Read-Only) |
| **$010000 – $010FFF** | 4 KB | **MMIO** | Mapped I/O: Keyboard, UART, Screen Buffer. |
| **$011000 – $FEFFFF** | ~15.8 MB | **RAM** | User Space (Programs & Data). |
| **$FF0000 – $FFFFFF** | 64 KB | **RAM** | **System Stack & Interrupt Vector Table (IVT).** |

### 5.2 Memory Mapped I/O (MMIO) Strategy

Since the instruction set is fixed at 16 bits, I cannot embed a full 24-bit physical address into a single instruction. Accessing MMIO requires a **Two-Step "Page & Offset" Process**:

1. **Set the Page:** Load the target Page Index (e.g., `$01` for I/O) into the Segment Register.
2. **Access the Offset:** Use standard Load/Store instructions.

**Example: Writing 'A' to Screen Buffer at `$010500**`

```asm
MOVI R2, $01       ; Load Page Index 0x01
MOV SEG, R2        ; Move to Segment Register
MOVI R3, $0500     ; Load Offset
MOVI R1, 'A'       ; Load Data
STORE R1, [R3]     ; Writes 'A' to Physical Address $010500

```

## 6. Bus Protocol (The Handshake)

### 6.1 Signal Line Definitions

| Signal | Width | Type | Function |
| --- | --- | --- | --- |
| **A0–A23** | 24 pins | Output | **Address Bus:** Defines target location. |
| **D0–D15** | 16 pins | Bi-Dir | **Data Bus:** Transmits payload. |
| **RW** | 1 pin | Output | **Read/Write:** High (1) = Read, Low (0) = Write. |
| **CE** | 1 pin | Output | **Chip Enable:** Active Low signal. |
| **CLK** | 1 pin | Input | **System Clock.** |

### 6.2 Electrical Logic Design

Control signals (CE, RESET, INT) utilize **Active-Low Logic** (triggered by Ground) for three reasons:

1. **Fail-Safe:** Pull-up resistors ensure the system defaults to "Inactive" if a wire breaks.
2. **Efficiency:** Sinking current (NPN logic) is faster/stronger than sourcing in TTL/CMOS.
3. **Wired-OR:** Multiple devices can safely pull the same Interrupt line Low simultaneously.

## 7. The Segmentation Unit (Hardware Implementation)

To convert internal 16-bit logic into 24-bit physical addresses without complex adders, I use **High-Byte Concatenation**.

### 7.1 The Mechanism

* **Pins A16–A23 (The Page):** Hardwired to the **Segment Register (8 bits)**.
* **Pins A0–A15 (The Offset):** Hardwired to the **Internal Address Bus (16 bits)**.

### 7.2 The Result

* **Hardware Benefit:** Requires **zero logic gates** for address generation.
* **Software View:** Memory appears as 256 distinct "Pages."

## 8. Interrupt Model

### 8.1 The Vector Table Location

*Correction from previous draft:* To allow dynamic driver loading, the **Interrupt Vector Table (IVT)** is located at the base of the High Memory Block in RAM, not ROM.

* **Location:** `$FF0000` (RAM).
* **Logic:** When Interrupt #X triggers, the CPU reads the address at `$FF0000 + (X * 2)`.

| Interrupt ID | Physical Address | Description |
| --- | --- | --- |
| **00** | `$FF0000` | **Divide by Zero** (Exception) |
| **01** | `$FF0002` | **NMI** (Critical Error / Power Fail) |
| **02** | `$FF0004` | **Timer Tick** (Context Switching) |
| **03** | `$FF0006` | **Keyboard / UART Input** |

### 8.2 The Hardware Handshake (Context Save)

Because the IVT is in a specific Segment ($FF), the hardware logic must switch pages automatically to find the handler.

1. **Request:** Device pulls **IRQ** Low.
2. **Context Save:** CPU pushes **FLAGS**, **SEG** (Current Segment), and **PC** to the Stack.
3. **Vector Fetch:**
* CPU forces internal Segment Register to **$FF**.
* CPU reads the 16-bit ISR address from the Table.


4. **Execute:** CPU jumps to the ISR.
5. **Return:** ISR executes `RETI`, restoring the old Segment and PC.

## 9. Privilege Model (System Protection)

### 9.1 Modes of Operation

To protect the OS, the CPU implements a "Dual-Mode" operation tracked by the **P-Flag**.

* **User Mode (P=0):** Cannot execute `CLI`/`STI` or write to the Segment Register.
* **System Mode (P=1):** Unrestricted access. The CPU automatically switches to System Mode when an Interrupt occurs.

### 9.2 The Status Register (FLAGS)

| Bit | Mnemonic | Name | Function |
| --- | --- | --- | --- |
| **15** | **P** | **Privilege** | 1 = System Mode, 0 = User Mode. |
| **14** | **I** | **Int Enable** | 1 = CPU responds to IRQs. |
| **13** | **S** | **Step** | 1 = Debug Mode. |
| **0-3** | **Z/C/N/V** | **ALU Flags** | Zero, Carry, Negative, Overflow. |
